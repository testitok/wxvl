#  Tenable 揭露“LeakyLooker”漏洞，将 Google Looker Studio 变成云端后门  
sec随谈
                    sec随谈  sec随谈   2026-03-16 01:08  
  
商业智能 (BI) 工具通常被视为“精美的镜子，以美观且可操作的图表形式反映组织的数据”。然而，Tenable 高级安全研究员 Liv Matan 的一项突破性调查显示，这些“镜子”可能变成攻击者的窗口。这组名为“LeakyLooker”的九个新型跨租户**漏洞**打破了该平台的基本承诺：“‘查看者’永远不应该能够控制、修改或窃取他们正在查看的数据”。  
  
这些漏洞（谷歌已经修复）暴露了谷歌云平台 (GCP) 环境中的敏感数据，可能会影响任何使用 BigQuery、Google Sheets、Spanner、PostgreSQL 等的组织。  
  
要了解 LeakyLooker 的严重性，就必须了解 Looker Studio 如何处理“信任”。该平台依赖于两种主要的凭证模型：  
  
  
- 所有者凭证：该报告使用所有者的权限来获取数据，允许他们与没有直接数据库访问权限的其他人分享见解。  
- 查看者凭证：该报告使用查看者自身的权限，确保他们只能看到他们明确被允许查看的内容。  
Tenable 的研究表明，这两条路径形成了“两个截然不同的‘信任边界’，可以分别受到攻击”。  
  
LeakyLooker 集群引入了一种新型攻击，可以窃取、插入甚至删除受害者环境中的数据。  
  
通过锁定所有者凭证，研究人员发现他们可以直接与报告的后端进行交互。  
  
  
  
Liv警告说： “攻击者通过向公共报告或共享报告发送精心构造的请求，触发 Looker Studio 服务，利用所有者的身份获取或操纵数据。 ”  
  
其中一个**漏洞**是别名注入，它利用了 Looker Studio 生成唯一列别名的方式。研究人员通过使用 SQL 注释 (/**/) 来“突破”这些用户控制的字符串，从而绕过过滤器，在受害者从未点击过任何链接的情况下，对所有者的整个 GCP 项目执行任意 SQL 查询。  
  
该研究还利用了 Looker Studio 的“实时数据”功能，该平台充当代理，将浏览器请求转换为实时 SQL 查询。其中一个**漏洞**利用了原生函数 (NATIVE_DIMENSION) 在报表加载时执行自定义 SQL。通过与受害者共享恶意报表，受害者会在不知情的情况下“在查看和加载共享报表时执行我们的恶意 SQL……甚至不会察觉”。  
  
  
  
这项研究中最巧妙的部分或许是用于窃取数据的“ping”机制。由于攻击者未经许可无法直接将窃取的数据插入到自己的表中，因此他们采用了跨租户日志分析的方法。  
1. 攻击者创建空的公共 BigQuery 表，表名以所有可能的字符命名（例如，exfil-a、exfil-l）。  
1. 当注入的脚本在受害者的数据中找到一个字符时，它会尝试从相应的攻击者表中读取该字符。  
1. “通过监控这些日志并拼凑出表访问的顺序，我们可以重建受害者的数据。”  
Tenable 的调查揭示了一个残酷的现实：全球部署最广泛的商业智能平台之一，可能已经成为入侵云基础设施核心的隐蔽入口。虽然谷歌已在全球范围内修复了这些问题，用户无需采取任何行动，但这项研究仍然对各组织敲响了警钟。  
  
  
  
管理员应始终“审核谁有权查看您的报告”，将 BI 连接器视为攻击面的关键部分，并且“不要允许服务访问您不再使用的连接器”。
